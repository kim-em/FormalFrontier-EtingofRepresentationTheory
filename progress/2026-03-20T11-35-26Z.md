### Core Blocker: `Decidable.casesOn` Dependent Type Rewriting

All remaining sorries are blocked by the same fundamental issue: composing `match`-based functions that case-split on `Decidable` instances.

When two functions both use `match inst i i, inst j i with | .isTrue _, .isFalse _ => fun e => e | ...`, their composition is definitionally `id ∘ id = id`. But Lean's kernel cannot reduce `match d, d' with ...` when `d` and `d'` are opaque applications (like `inst i i`). Standard tactics fail:
- `rw` / `▸`: "motive is not type correct" (dependent types)
- `simp`: doesn't reduce `match` expressions
- `split`: "no match expression found" (nested inside applications)
- `cases`: "generalize failed" (dependent types)
- `match` in proof: works for proving equalities on arrow-level (as in `origArrow_arrowsInto_to_arrowsOutReversed`), but fails for Sigma-level equalities because the Sigma type itself contains `Decidable.casesOn` that must also reduce

The first round-trip (`origArrow_arrowsInto_to_arrowsOutReversed`) works because after `unfold + simp only [id]`, the goal is a simple arrow equality where `match` in the proof can directly reduce both sides. The second round-trip (`arrowsOutReversed_arrowsInto_roundtrip`) fails because it's a Sigma equality where the TYPE of the second component contains `Decidable.casesOn` that doesn't reduce.

### Recommended Fix

The root cause is that `reversedArrow_eq_ne` in `Definition6_6_3.lean` uses `match` while the type `ReversedAtVertexHom` also uses `@Decidable.casesOn`. These are the same compiled form, but their composition through Sigma types creates irreducible terms.

**Option A**: Redefine `ReversedAtVertexHom` to NOT use `@Decidable.casesOn` at the type level. Instead, use a plain definition with explicit cast lemmas. This would make all type transports explicit and composable.

**Option B**: Define a custom `Equiv` for `ArrowsOutOf Q̄ᵢ i ≃ ArrowsInto Q i` using `Equiv.ofBijective` with a direct injectivity/surjectivity proof (avoiding the round-trip).

**Option C**: Add a rewrite lemma to `Definition6_6_3.lean` that states `reversedArrow_eq_ne hb (arrowInto_to_reversed hb e) = e` where `arrowInto_to_reversed` is a new inverse function defined there.
