### Key finding: the diamond was a red herring

The issue body anticipated a `Module ↥B (↥(S iLam) →ₗ[A] E)` instance
diamond and recommended four resolution approaches in order (`letI`
pinning → heartbeat bump → explicit `LinearEquiv.ofBijective`
annotation → refactor `isSimpleModule_homA_centralizer`).

When I attempted approach 1 (`letI : Module ↥B (↥(S iLam) →ₗ[A] E) :=
centralizerModuleHom` at the top of the proof), the build hit a
1.2M-heartbeat timeout much earlier in the proof than expected, with
the local context showing `this✝ : Module ... := sorry` (Lean's
pretty-printing for elaboration-failed terms). Removing the `letI` and
just bumping heartbeats to 6.4M / 3.2M revealed the actual blocker:

```
error: failed to synthesize instance of type class
  MulActionHomClass (β iLam →+ DirectSum ι fun i ↦ β i) k
    (β iLam) (DirectSum ι fun i ↦ β i)
```

The WIP body's `hα_block` proof used a generic `map_smul` to commute
`α • _` past `DirectSum.of β iLam`, but `DirectSum.of` is an
`AddMonoidHom`, not a `LinearMap` — there is no `MulActionHomClass`
instance. The fix is the specific lemma `DirectSum.of_smul`
(`Mathlib.Algebra.DirectSum.Module:93`), which is `(lof R ι M i).map_smul`
in disguise.

With the one-line `map_smul → DirectSum.of_smul` change, the body
compiles cleanly under the original heartbeat profile (6.4M / 3.2M).
No `letI` pinning was needed; no diamond actually existed at the
`IsSimpleModule.congr Ψ.symm` site, because the global
`centralizerModuleHom` is the only `Module ↥B (↥(S iLam) →ₗ[A] E)`
path Lean ever sees throughout the proof.
