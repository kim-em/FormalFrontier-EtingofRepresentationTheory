## Accomplished

Claimed #2431 (Refactor dTildeDim for definitional reduction). Performed a
thorough reducibility analysis to determine the feasibility of the issue's
literal goal. Empirically tested multiple candidate formulations for
`dTildeDim` in a scratch block within the target file (all reverted — no
code changes committed).

**Finding**: the issue's goal — rfl/dsimp-level reduction of
`(dTildeRep k m).obj v` at k-dependent vertices (`⟨k+3, _⟩`, `⟨k+4, _⟩`,
`⟨k+5, _⟩`) — is **fundamentally unreachable** with a `Fin (k+6)` vertex
type. Any leaf-vs-interior classification requires a `Nat.decLe` /
`Nat.decEq` / `Nat.sub` comparison between two k-dependent values. These
reduce via stripping `succ` constructors from both sides until one side
reaches `zero`; when both sides are of the form `k + c`, stripping stops
at the free variable `k`, which matches neither `zero` nor `succ _`. So
`Nat.decLe (k+3) (k+3) → Nat.decLe k k` is stuck.

Approaches tested and rejected:
- Nested `Fin.cases` + `Fin.lastCases` — `Fin.lastCases` uses a `by simp`
  cast proof inside `Fin.reverseInduction.go`, never reduces by rfl.
- `Fin.rev` to access right leaves by peeling from the left — `Fin.rev`
  itself doesn't reduce at k-dependent vals (`(k+5) - (k+5)` stuck).
- Subtraction tricks (`k+3 - v.val`, `v.val - (k+4)`, etc.) — any
  `Nat.sub` involving k on the second arg gets stuck.
- `@[simp]` lemmas for dim at branch points — provable by
  `simp [dTildeDim]; omega`, but not rfl-provable, so `dsimp only` can't
  use them; `simp only` rewrites introduce casts (no improvement over
  the existing `▸ ▸` bridges).
