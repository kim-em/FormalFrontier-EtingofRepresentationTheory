## Accomplished

Issue #2633 (partial): Schur-Weyl L_i `Theorem5_18_4_GL_rep_decomposition_simple`.

Landed in this session: a generic simplicity-transfer helper
`isSimpleModule_monoidAlgebra_GL_of_centralizer_simple` in
`EtingofRepresentationTheory/Chapter5/SchurWeylGLTransfer.lean`. Given a
`diagonalActionImage k V n`-simple module `M` with `V = Fin N → k` and a
`GL_N` representation `ρ : Representation k GL_N M` whose action factors
through `g ↦ ⟨g^⊗n, _⟩ ∈ diagonalActionImage`, the helper concludes
`IsSimpleModule (MonoidAlgebra k GL_N) ρ.asModule`.

The proof packages:

* `Submodule k[GL_N] ρ.asModule` ↪ `Submodule k M` via `restrictScalars`,
* GL_N-closure ⇒ `(End k V)ˣ`-closure via `Matrix.GeneralLinearGroup.toLin`,
* `(End k V)ˣ`-closure + simplicity ⇒ `⊥`/`⊤` via the existing
  `submodule_eq_bot_or_top_of_unit_smul_mem` (from #2611),
* `⊥`/`⊤` of `restrictScalars` ⇔ `⊥`/`⊤` of original via
  `restrictScalars_eq_bot_iff` / `_top_iff`.

The helper's hypothesis `h_act` is the natural action-factorisation
condition: every `(L i).ρ g` post-composes by `g^⊗n` (an element of
`diagonalActionImage`) on the multiplicity hom-space.

Build is green: zero new sorries, `lake build SchurWeylGLTransfer`
passes.

## Current frontier

Sub-issue #2633 (the user-facing decomposition theorem
`Theorem5_18_4_GL_rep_decomposition_simple`) is **not** landed in this
PR. The helper is the missing piece of infrastructure that enables it.

The remaining work is to thread the helper through the chain
`Theorem5_18_4_bimodule_decomposition_explicit` → glHom → ρ_i →
`FDRep.of (ρ i)` and produce the existential. Three integration
challenges encountered during the attempt:

1. **Typeclass diamond on `Module (centralizer A) (S' i →ₗ[A] V^⊗n)`**:
   `bimodule_decomposition_explicit`'s output uses one specific path
   through `centralizerModuleHom` (instantiated when the existential is
   constructed); a fresh `haveI` of the same type elaborates with a
   different instance path (e.g. `(symGroupImage).moduleLeft` vs.
   `Submodule.module`), so the resulting `IsSimpleModule` term does not
   unify with the locally-elaborated type.

2. **`Module (diagonalActionImage)` on the multiplicity hom-space**:
   the `centralizerModuleHom` instance lives on the centralizer
   subalgebra; transporting via `h_eq.symm.le` (using
   `Module.compHom (Subalgebra.inclusion h_eq.symm.le).toRingHom`)
   gives a computational smul that satisfies `h_act` by `rfl`, but
   `IsSimpleModule` transfer via `LinearMap.isSimpleModule_iff_of_bijective`
   still hits the typeclass-search problem in (1).

3. **`(L i).ρ.asModule` vs. `Representation.asModule (L i).ρ`**: dot
   notation on `(L i).ρ : GL_N →* (L i).V →ₗ[k] (L i).V` (a `MonoidHom`)
   does not pierce the `Representation` `abbrev` to find `.asModule`;
   need to write `Representation.asModule (L i).ρ` explicitly.
