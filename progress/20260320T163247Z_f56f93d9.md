### Blocker analysis: Decidable.casesOn opacity

All 4 remaining sorries in Proposition6_6_6.lean share the same root cause: `reversedArrow_eq_ne` is defined by `match inst i i, inst j i with` (producing `Decidable.casesOn`), while the other side uses `cast` with a proof. The standard approach (`unfold + rw [h_di, h_dj]`) fails because:

- `rw` fails with "motive is not type correct" because `inst i i` appears in dependent type positions (inside `Decidable.rec` in the quantified variable's type AND inside `match` in the body). The abstraction `fun _a => ∀ (e : Decidable.rec ... _a), (match _a, ... with ...) e = cast ⋯ e` can't be typed because the match body references `ReversedAtVertexHom Q i i j` (not unfolded) while the binder type is `Decidable.rec ... _a` (unfolded).

- `cases inst i i` fails with "generalize failed" for the same reason — it can't abstract `inst i i` from the dependent positions.

- `subst` fails because `i` occurs in the context through `hi : IsSink Q i`.

**Approaches tried and failed:**
1. `unfold + rw [h_di, h_dj]` — motive not type correct
2. `unfold + cases inst i i` — generalize failed
3. Match-based inverse (`origArrow_to_reversedHom`) — same unfold+rw issue for round-trips
4. Adding `reversedAtVertex` to unfold list — same issue
5. `conv` targeting specific occurrences — same fundamental problem

**Potential resolution paths:**
1. **Redefine `reversedArrow_eq_ne` in Definition6_6_3.lean** to use `cast` internally instead of `match`. This would make the bridge trivial but requires updating `reflFunctorPlus_mapLinear_eq_ne` which depends on the current match-based definition.
2. **Use native term-level `Decidable.recOn`** to write the proof without tactics, avoiding the motive construction issues. This is extremely verbose but may work.
3. **Dimension argument** for `ker ≤ range`: bypass the bridge entirely by proving `dim(range(ψ)) = dim(ker(Φ))` using rank-nullity, then conclude from `range(ψ) ≤ ker(Φ)`.
