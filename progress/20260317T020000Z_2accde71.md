### 1. Reflection Analysis

**What worked well in Phase 1:**
- **Validation-first pattern**: Creating schemas + validators before content work (issues #414, #451, #459) prevented format drift across agents and caught errors early. Every validation tool was reused 5+ times.
- **Batch PRs over atomic PRs**: The most productive sessions batched 8-17 pages per PR. Single-page PRs had disproportionate coordination overhead.
- **CONVENTIONS.md as quality driver**: Transcription accuracy was excellent (1 error in 60+ spot-checked pages) because conventions were documented early and followed consistently.
- **Parallel structure analysis by chapter**: Chapters are naturally independent units. Distributing 1 chapter per agent with no artificial dependencies worked perfectly.
- **Progress file protocol**: Every session produced clear handoffs. No agent was ever confused about project state.

**What caused friction:**
- **Linear dependency chains for parallel work**: Issues #255, #256, #259, #262 each had 3-6 claim/skip/release cycles because of unnecessary serial dependencies on transcription pages.
- **Stale `has-pr` labels**: When PRs were closed without merging, their labels remained, blocking other agents. Manual cleanup was needed (issue #434 released 11 stale claims).
- **Late cross-validation**: Chapter 5 gap discovery (9 missing pages) happened during cross-validation, not during original structure analysis. This blocked items.json assembly.
- **Stale branches**: PRs #335, #342, #343 accumulated merge conflicts and required manual content salvage.

### 3. Issue Sizing Assessment

Analyzed 30 merged PRs for scope effectiveness:

| Category | Examples | Size | Verdict |
|----------|---------|------|---------|
| Tooling creation | #414, #451, #459 | 200-350 lines, 3 files | Well-scoped — completed in single turns |
| Structure analysis (1 chapter) | #426, #430, #436 | 500-3300 lines, 2-44 files | Good scope — scales with chapter size |
| Batch transcription | #438, #407 | 220-265 lines, 9-11 files | Good scope |
| Reviews | #449, #456, #437, #469 | 25-125 lines, 1-2 files | Well-scoped — lightweight, focused |
| Assembly (multi-stage) | #460 | 11,700 lines, 589 files | Borderline large — completed but could benefit from splitting |
| Conservative deps generation | #465 | 179,800 lines, 593 files | Generated output — size is fine since it's scripted |
| Meditate | #421 | 205 lines, 3 files | Good scope |

**Recommendations:**
- Assembly-type issues that touch 500+ files should be validated for context window pressure. If the issue requires reading many files *and* generating output, consider splitting into "generate" + "validate" sub-issues.
- Review issues are appropriately lightweight — keep them at 1-2 files of output.
- Phase 2 chapter-level dependency analysis issues should be well-scoped since each chapter's blobs are independent.
