## Accomplished

Investigated and documented the technical blocker for Proposition 6.6.6 (reflection functor inverse). Added infrastructure:

1. **`transport_iso` helper**: Lemma that transports a `QuiverRepresentation.Iso` across a quiver instance equality. Proved by `subst h; exact iso`. This cleanly handles the `Eq.mpr` opacity when both representations are on the same quiver.

2. **Documented mathematical arguments** for both theorems:
   - **Sink case** (F⁻ᵢ F⁺ᵢ V ≅ V): At vertex i, the composition gives coker(ker(φ) ↪ ⊕V_j). By the first isomorphism theorem, ⊕V_j / ker(φ) ≅ im(φ) = V_i (since φ is surjective).
   - **Source case** (F⁺ᵢ F⁻ᵢ V ≅ V): At vertex i, the composition gives ker(⊕V_j → coker(ψ)). From the short exact sequence 0 → V_i →ψ ⊕V_j → coker(ψ) → 0, the kernel ≅ im(ψ) ≅ V_i (since ψ is injective).

3. **Identified the fundamental blocker**: `transportReversedTwice` uses `▸` (Eq.mpr) to rewrite the Quiver instance on `QuiverRepresentation`. This creates opaque terms that prevent reasoning about `.obj v` and `.mapLinear`. Constructing a `LinearEquiv` between vertex spaces of representations on different Quiver instances triggers Lean 4's "synthesized type class instance is not definitionally equal to expression inferred by typing rules" error. This happens because `QuiverRepresentation` uses `[instAddCommMonoid]` and `[instModule]` as instance-marked fields, and the synthesis check picks up the ambient Quiver instance rather than the intended one.

## Current frontier

Both theorems still have `sorry`. The mathematical content is clear but the dependent type transport is blocked. Three potential resolutions:

(a) **Restructure `QuiverRepresentation`** to use `{}` (implicit) instead of `[]` (instance) fields for `instAddCommMonoid` and `instModule`. This removes the synthesis check. Requires updating all callers (Definition 6.6.3, 6.6.4).

(b) **Redefine `transportReversedTwice`** to explicitly construct the representation field-by-field using `ρ.obj` directly. The `[instAddCommMonoid]` synthesis check prevents this with the current structure definition.

(c) **Use a heterogeneous Iso** that doesn't require transporting across quiver instances. The `equivAt` field would use `@LinearEquiv` with explicitly provided module instances. But even `@LinearEquiv` triggers the synthesis check.

Option (a) is the cleanest but has the largest blast radius.
